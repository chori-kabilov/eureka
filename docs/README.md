# EUREKA — CRM для учебного центра (MVP)

EUREKA — система управления учебным центром: студенты, преподаватели, курсы, группы, расписание, уроки, посещаемость и платежи.

Цель V1: запустить в реальном учебном центре и закрыть ежедневные операции без хаоса в таблицах/чатах.

---

## 1) Что входит в MVP

### Модули (MVP)
1. **Студенты**
   - учёт студентов
   - привязка к группам/курсам
   - просмотр своего расписания (если есть личный кабинет)
2. **Преподаватели**
   - учёт преподавателей
   - расписание преподавателя
   - отметка посещаемости по урокам
3. **Курсы**
   - каталог курсов
   - правила курса (оплата/пропуски/оплата преподавателя)
4. **Группы**
   - создание групп по курсам
   - назначение преподавателя
   - расписание группы (дни недели + время)
   - период обучения (дата начала/окончания)
5. **Календарь / Расписание**
   - список уроков по дням/неделям
   - перенос времени/отмена урока
   - замена преподавателя на конкретный урок
6. **Посещаемость**
   - отметка присутствия/отсутствия для каждого студента на каждом уроке
7. **Платежи (без долгов)**
   - запись факта оплаты (оплачено)
   - история оплат
   - простые отчёты по оплатам

---

## 2) Что НЕ входит в MVP (жёстко)

- частичные оплаты и “ожидающие платежи”
- автоматическая блокировка за неуплату
- онлайн-обучение и видеоплатформа
- полноценный чат-мессенджер внутри системы
- сложная аналитика, графики, BI
- гибкий конструктор ролей и прав
- темы/оформление
- продвинутые уведомления (email/sms/telegram) как обязательная часть
- библиотека материалов как полноценное хранилище (только V2, если потребуется)

## 3) Главные принципы системы

### 3.1 Базовая единица — УРОК
Система строится вокруг урока, потому что:
- посещаемость ставится на урок
- замены преподавателя происходят на урок
- фактическая учебная деятельность фиксируется уроками

### 3.2 Курс = правила
Курс — это не просто название и цена. У курса есть правила:
- **Тип оплаты студента**: за урок / за месяц / за весь курс
- **Правило пропусков**: по умолчанию “сгорает” (в MVP переносы не делаем)
- **Оплата преподавателя**: за урок или за час
- **Ограничения**: можно ли студенту учиться параллельно в других курсах (опционально)

### 3.3 Платежи без долгов
В MVP нет понятия “студент должен”.
Есть только факт:
- **Оплата внесена** (за месяц/курс/пакет)
- либо **оплаты нет**

Контроль “почему не оплатил” остаётся организационным процессом, не логикой системы.

## 4) Роли и доступы (MVP)

### Администратор
- полный доступ ко всем сущностям
- создаёт курсы/группы/расписание
- фиксирует оплаты
- подтверждает замены преподавателей

### Преподаватель
- видит только свои группы/уроки
- отмечает посещаемость на уроках
- видит свои начисления (если включено в V1, иначе — V2)

### Студент
- видит своё расписание и посещаемость
- редактирует только свой профиль (минимум)

### Родитель
- видит привязанных детей (профиль/расписание/посещаемость)
- оплата в MVP фиксируется администратором (родительский кабинет — “просмотр”)

---

## 5) Основные сценарии (MVP)

1) **Создать курс**
- администратор создаёт курс и задаёт правила (тип оплаты, пропуски, оплата преподавателя)

2) **Создать группу**
- выбрать курс
- назначить преподавателя
- задать период (даты)
- задать расписание (дни/время)

3) **Записать студента в группу**
- выбрать студента
- выбрать группу
- добавить в состав группы

4) **Сформировать уроки в календаре**
- из расписания группы система формирует уроки
- администратор может переносить/отменять отдельные уроки

5) **Провести урок и отметить посещаемость**
- преподаватель открывает урок
- отмечает студентов: присутствовал/отсутствовал

6) **Замена преподавателя**
- администратор указывает заменяющего на конкретный урок
- урок считается проведённым заменяющим преподавателем

7) **Оплата**
- администратор фиксирует оплату:
  - кто оплатил (студент/родитель)
  - за что (месяц/курс/пакет)
  - сумма и дата
- в системе нет “долга”, только факт оплаты и история

---

## 6) Быстрые договорённости по MVP

- MVP считается готовым, когда:
  - администратор может вести группы и расписание
  - преподаватель может отмечать посещаемость
  - оплаты фиксируются без долгов и без “ожиданий”
  - отчёт по оплатам за период можно получить

- Любая новая идея проходит фильтр:
  1) помогает ли это ежедневно администратору/преподавателю?
  2) без этого нельзя запустить V1?
  Если ответ “нет” — в V2.
